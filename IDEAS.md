## AI 建议先放进来的想法

- **场景摘要链**：每次决议结束后，自动将本轮成品提炼成 2-3 句"章节摘要"写入 memory.json，供下一轮委员会参考，解决跨会话记忆断层问题。可能的价值：让长篇写作不依赖完整上文，大幅降低 token 消耗。可能的难点：摘要质量不稳定，过度压缩可能丢失关键细节。

- **伏笔追踪器**：在 memory.json 中增设 `foreshadowing` 字段，一致性委员每轮负责登记"已埋伏笔"和"已回收伏笔"，主动提醒哪些伏笔悬而未决。可能的价值：解决长篇最难的伏笔管理问题，这是单 AI 写作最容易失控的地方。可能的难点：委员识别伏笔的准确率依赖 prompt 质量。

- **委员个性参数化**：给每位委员配置"性格倾向"字段（如批评家可设为"严苛派"或"建设派"），影响其 system prompt 措辞，让同一角色在不同项目中表现出不同风格。可能的价值：让委员会更有"个性"，辩论更真实，也为 YAML 可配置打基础。可能的难点：参数过多反而难以调试，需要找到最小可控集合。

- **风格锚点文本**：会长可上传一段参考文本（如喜欢的作家片段），润色师和作家在创作时以此为风格参照，而不只是靠文字描述风格。可能的价值：解决"风格描述不精确"问题，直接示范比文字说明更高效。可能的难点：如何将参考文本有效注入 prompt 而不占用过多 context。

- **多稿对比视图**：在前端右栏支持"初稿 / 修订稿 / 润色稿"三版并排或 diff 对比，让会长清楚看到委员会每一步改了什么。可能的价值：提升透明度，会长能判断哪一步的修改是好的、哪一步是退步。可能的难点：纯 HTML/JS 实现 diff 渲染需要引入小型库。

- **会长批注转指令**：会长在成品上的选中改写操作，自动归纳成可复用的"改写规则"（如"对话要更简短"），可选择是否应用到下一轮全局 prompt。可能的价值：让局部修改沉淀为创作偏好，而不是每次手动重说。可能的难点：从单次修改中归纳规则容易过拟合。

- **辩论热度标注**：在中栏辩论流中，对委员分歧最大的段落自动高亮标注"争议点"，并在成品右栏对应位置显示小标记，让会长一眼看出哪里是"委员会最纠结的地方"。可能的价值：帮助会长快速定位需要重点关注的内容，而不是通读全部辩论。可能的难点：判断"分歧大"需要额外逻辑，可能误判。

- **读者代表委员**：增加第 6 席"读者代表"，角色是一个特定类型的目标读者（如"轻小说读者"或"硬科幻爱好者"），在批评环节从受众角度发言，不讨论技法只讨论阅读体验。可能的价值：弥补现有委员偏"创作视角"的盲区，更接近真实读者反馈。可能的难点：读者代表的 prompt 容易写得太泛，意见价值低。

- **委员发言权重配置**：允许会长调整各委员意见在最终拍板时的权重（如"本轮侧重一致性，批评家意见权重降低"），主编综合时按权重取舍。可能的价值：让会长对每一轮的侧重点有更细粒度的控制，不必每次都平权决策。可能的难点：权重如何注入 prompt 逻辑上有一定复杂度。

- **离线导出包**：一键将当前作品打包为独立 HTML 文件（含辩论记录、各轮成品、修改历史），可脱离服务器在浏览器直接阅读。可能的价值：便于分享、归档、展示给他人，也是开源项目的好演示材料。可能的难点：内嵌所有数据的单文件 HTML 体积可能较大。

- **角色编辑器**：结构化角色卡界面，字段包括姓名、外貌、性格、背景、目标、说话口吻、与其他角色的关系；会长可自由增删自定义字段；保存后自动写入 memory.json，委员会每轮创作前自动参考。可能的价值：解决角色一致性最根本的问题——委员有明确依据可查，而不是靠 prompt 里几句描述。可能的难点：角色卡字段设计要足够灵活又不过度复杂，自定义字段如何序列化需要设计。

- **世界观编辑器**：按类别组织世界设定条目，类别包括地点、势力 / 组织、规则体系（魔法 / 科技 / 社会制度）、大事年表、术语表；每条为独立条目，会长可增删类别；一致性委员每轮自动引用相关条目校对。可能的价值：让世界观从散落在 setting.json 的文字堆变成结构化知识库，委员能精准查阅。可能的难点：条目之间的关联关系（如某地点属于某势力）如何建模。

- **顶部 Tab 工作区切换**：前端顶部设"设定 / 创作 / 历史"三个主工作区 Tab，设定区管角色 + 世界观编辑器，创作区是三栏辩论界面，历史区浏览所有历史轮次；避免所有功能挤在一个页面。可能的价值：信息架构清晰，不同任务有专属空间，减少界面噪音。可能的难点：Tab 切换时需保持创作区 WebSocket 连接不断。

- **辩论流增强**：中栏辩论流支持折叠单条委员发言、固定（pin）关键意见置顶、按委员身份筛选显示；长辩论时不需要滚动全文寻找某人发言。可能的价值：辩论流是本项目核心体验，越可读越能体现委员会价值。可能的难点：pin 逻辑与实时滚动共存时 UI 状态管理较复杂。

- **成品区增强**：右栏成品展示区增加字数统计 + 预计阅读时长（按 300 字/分钟）；段落级别显示"本段经过几轮修订"小标记；选中文本时弹出快捷操作菜单（局部改写 / 复制 / 标注）。可能的价值：让会长对成品有更丰富的感知，选中改写的入口更顺手。可能的难点：段落修订次数追踪需要与 debate_log 关联。

- **快捷指令面板**：左栏本轮指令输入框旁增加预设模板库，内置常用场景指令（"写一段正面冲突"、"推进感情线"、"揭示一个秘密"、"过渡场景"），点击一键填入，会长也可保存自定义模板。可能的价值：降低每轮输入成本，减少指令措辞不准导致的产出偏差。可能的难点：内置模板要足够通用，避免预设太强绑定特定题材。

- **历史回溯视图**：历史工作区列出所有历史轮次（时间 + 本轮指令摘要），点击任意轮次可查看当时的完整辩论记录和成品快照；支持将历史版本的成品"恢复"为当前成品。可能的价值：会长反悔时有退路，也便于复盘哪一轮的委员会表现最好。可能的难点：revisions/ 目录的数据结构需要支持完整快照而不只是 diff。

- **键盘快捷键**：为高频操作绑定快捷键：开始本轮（Ctrl+Enter）、停止生成（Esc）、聚焦指令输入框（Ctrl+L）、触发局部改写（选中文本后 Ctrl+E）；在界面角落显示快捷键速查卡。可能的价值：重度使用时大幅提升效率，体现产品专业感。可能的难点：需避免与浏览器默认快捷键冲突。

- **深色模式**：前端支持深色 / 浅色主题切换，默认跟随系统设置；长篇写作场景下深色模式护眼且更沉浸。可能的价值：长时间使用的舒适度，也是开源项目用户期待的基础功能。可能的难点：纯 CSS 变量实现主题切换不难，但要保证辩论流的委员颜色在深色下依然可读。

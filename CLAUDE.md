# AI 作家委员会（AI Writer Council）

> 本文件是项目"宪法"。任何接手者（包括 AI）请先通读再动手。

---

## 项目目标

通过多 agent 分工 + 结构化记忆，让 AI 产出长篇小说，验证"委员会式"创作能否优于单 AI 写作。最终目标是 10–20 万字的长篇小说。

---

## 当前阶段：MVP v2

**MVP 的唯一验证目标**：跑通"一次决议轮次"的完整流程。证明委员会协作产出 > 单 AI 直接产出。

- 不是写完 1 万字小说
- 是跑通一次完整的委员会协作链路

---

## 核心设计原则（不可妥协）

1. **前后端分离**：后端管"权力配置"（委员、流程、模型），前端只是"会长操作台"
2. **模型可替换**：所有 LLM 调用走统一接口 `call_llm(role, messages, model)`，不绑定特定厂商
3. **委员可配置**：未来通过 YAML 定义委员（MVP 阶段先硬编码 5 席，但代码结构要为可配置预留）
4. **局部修改优于全局重生成**：会长能选中文本片段精确改写，不浪费 token、不破坏好内容
5. **会长不被打扰**：纯自动跑流程，会长只在开头设定 + 结尾审核 + 主动局部修改时介入

---

## 技术栈

| 层次 | 技术 |
|---|---|
| 后端 | Python + FastAPI |
| 前端 | 纯 HTML + CSS + JavaScript（不用 React/Vue，保持轻量） |
| 实时通信 | WebSocket（实时展示辩论流） |
| 数据存储 | JSON 文件（MVP 阶段） |
| LLM | MVP 默认 Claude Sonnet 4.6，接口抽象支持替换 |

---

## 部署形态

本项目为**纯本地应用（Local-First）**：

- 后端 + 前端均在用户本地电脑运行
- 用户访问 localhost 使用，不连接项目作者的任何服务器
- 所有数据（API key、作品、记忆、配置）存储于用户本地文件系统
- 项目本身不收集任何用户数据，不发遥测，不"phone home"

## MVP 委员会构成（5 席）

| 委员 | 职责 |
|---|---|
| 主编 | 制定本轮场景大纲，调度其他委员，最终拍板 |
| 作家 | 实际写作 |
| 批评家 | 提情节 / 节奏 / 逻辑意见 |
| 一致性委员 | 校对人物 / 世界观 / 伏笔有无矛盾 |
| 润色师 | 最后语言打磨 |

**未来扩展位预留**：风格委员、情感委员、节奏委员、读者代表

---

## MVP 模型分配方案

| 委员 | 默认模型 | 理由 |
|---|---|---|
| 主编 | Claude Sonnet 4.6 | 调度 + 拍板，需要好推理 |
| 作家 | Claude Sonnet 4.6 | 文笔决定下限，必须用最强 |
| 批评家 | Claude Sonnet 4.6 | 挑刺需要敏锐 |
| 一致性委员 | DeepSeek V4 | 校对任务，不需要顶级文笔，可大幅省成本 |
| 润色师 | DeepSeek V4 | 润色相对模式化，可大幅省成本 |

Gemini 3 Pro 作为备用模型，初期不分配岗位，但 `call_llm` 接口必须支持。
所有模型分配通过配置文件控制，会长可在前端"设置"页修改。

---

## MVP 工作流（一次决议）

1. 会长输入：作品设定（题材 / 角色 / 世界观 / 风格 / 禁忌）+ 本轮指令 + 期望长度（500–5000 字可调）
2. 主编出场景大纲（3–5 句话）
3. 作家写初稿
4. 批评家提意见 + 一致性委员检查
5. 作家修订（最多 3 轮辩论）
6. 主编拍板
7. 润色师打磨
8. 成品交付，前端实时展示全过程

---

## MVP 前端（三栏布局）

- **左栏**：作品设定 + 本轮指令 + 长度滑块 + 开始按钮
- **中栏**：辩论实时流（WebSocket 推送）
- **右栏**：成品展示 + **选中文本即可局部修改**（核心功能）

---

## MVP 边界——明确不做

- 不做多场景串联
- 不做章节摘要链
- 不做用户认证
- 不做云端部署
- 不做 YAML 委员自定义（先硬编码）
- 不做多模型混搭（先全 Claude）

## API Key 管理与隐私原则（强制）

1. **会长在前端"设置"页填写各模型的 API key**
2. **Key 通过 localhost 传给后端，后端写入本地 .env 文件**
3. **.env 永远不进 git 版本控制**（.gitignore 必须包含 .env）
4. **所有模型调用走统一接口** `call_llm(role, messages, model_config)`
5. **model_config 必须支持自定义 base_url**——用户可填官方或自选第三方代理
6. **代码和文档永不内置任何第三方代理服务的具体 URL 或名称**——隐私和合规考虑
7. **API key 绝不写入日志**
8. **API key 绝不发送给本项目作者或除用户主动配置的 AI 厂商之外的任何第三方**
9. **首次启动如检测到无 .env，引导会长进入设置页填写**
10. **提供 .env.example 模板**，仅展示字段名和官方 endpoint 占位符

`.env.example` 必须包含以下结构（供实现时参考）：

```bash
# Claude (Anthropic)
ANTHROPIC_API_KEY=your_key_here
ANTHROPIC_BASE_URL=https://api.anthropic.com  # 可替换为自选代理，留空则用官方

# DeepSeek
DEEPSEEK_API_KEY=your_key_here
DEEPSEEK_BASE_URL=https://api.deepseek.com    # 可替换为自选代理，留空则用官方

# Gemini（备用，暂不分配委员岗位）
GEMINI_API_KEY=your_key_here
GEMINI_BASE_URL=https://generativelanguage.googleapis.com  # 可替换为自选代理，留空则用官方
```

## 项目协作约定（会长与 AI 之间）

- **会长**（令）是产品负责人，AI 是技术执行
- 重大设计决策必须先讨论再动手，不要擅自扩展功能
- 所有功能想法先记到 `IDEAS.md`，不直接实现
- 写代码前先讲思路，会长同意后再写
- 用中文回复，代码注释也用中文
